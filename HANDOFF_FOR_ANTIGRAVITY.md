## 2. 最近完成的高风险/核心修改 (切勿覆盖)

### A. 后端群发匹配 Bug 修复 (`web_server.py`)
*   **问题**：前端任务提交的群组 ID 常常为正数 (如 `2291114263`)，而数据库中保存 of ID 常常带有负号和前缀 (如 `-1002291114263`)。原代码直接使用 `GroupDb.id == str(chat_id)` 导致所有群组类型匹配失效，全部退化为默认的“英文短”。
*   **修改**：
    *   在 `campaign_worker_task` 循环开始前初始化了 `final_msg = ""`。
    *   在查询群组类型时，使用 `in_([str(chat_id), f"-{chat_id}", f"-100{chat_id}"])` 同时兼容正数、负数、`-100` 前缀，对 `GroupDb` 和 `ScrapedGroupDb` 进行动态穿透查询。
    *   **以数据库最新的分类为准**：取消了之前仅在任务不存在时查询的限制，改为每次发送前都实时去底层表拉取最新类型，从而实现用户在“群组列表”修改类型后，运行中的任务能实时感应并发送对应的广告词。
*   **成功发送日志**：
    *   在写入 `CampaignLogDb` 时，若发送成功，将广告词拼接在 `detail` 字段后面，格式为：`[预览: {final_msg}]`。

### B. 前端“单次群发投递流水明细”表格修改 (`index-C5s22-6-.js`)
*   **修改内容**：
    *   **增加列**：在“状态”列和“流水详情”列之间，新增了一列 **“发送文本”**（表头占比调整为：状态 10%、发送文本 22%、流水详情 16%）。
    *   **Hover 预览**：新增的“发送文本”列从 `e.detail` 中提取 `[预览: ...]` 部分，默认截断展示前 25 个字符。当鼠标悬浮在文本上时，利用 HTML 的 `title` 属性弹出完整广告词气泡。
    *   **流水详情净化**：原本的“流水详情”列自动去除了 `[预览: ...]` 尾巴，只显示干净的“消息发送成功”或失败原因。

### C. 启动与停止速度优化 (`rosepay.service`)
*   **修改**：在 systemd 服务配置中添加了 `TimeoutStopSec=3`，并在后端使用了并发自愈上线。目前服务重启仅需约 7 秒，且重启后能瞬间自动接管之前处于 `running` 状态的任务，实现平滑热部署，对正在运行的轰炸任务零影响。
